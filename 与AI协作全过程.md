### 🤖 AI 智能体 (Assistant)

**Claude Code** 是 Anthropic 最近刚推出的一个**纯命令行（CLI）形态的 AI 编程助手**。

既然你已经在用 Cursor，你可能会好奇：“Cursor 的 Agent 已经很强了，为什么还需要终端里的 Claude Code？” 简单来说，它的核心价值和使用场景与 Cursor 是互补的。

以下是 Claude Code 的三大核心用处和它的“杀手锏”：

---

### 1. 它是一个“有手有脚”的终端自动化代理 (Agent)
你在 Cursor 的侧边栏跟我聊天，我虽然能帮你写代码、甚至执行一些终端命令，但我本质上还是依赖 Cursor 编辑器的外壳。
而 Claude Code 直接生活在你的终端（Terminal）里。**它最大的特点是“高度自治”**：
- **场景**：你可以在终端里输入 `claude "帮我排查一下为什么 npm run build 失败了，并直接修复它"`。
- **它的动作**：它会自己去读错误日志 -> 搜索你的代码库 -> 找到错误文件 -> 自动修改代码 -> **自己再次运行 `npm run build` 测试** -> 发现新错误继续修 -> 直到 build 成功，最后向你汇报。
- **用处**：非常适合用来做环境配置、依赖安装排错、自动化运行测试脚本等“脏活累活”。

### 2. 脱离 IDE 的轻量级编程
不是所有的修改都需要打开一个庞大的 IDE。
- 如果你只是想 ssh 连上一台没有装 Cursor 的**远程 Linux 服务器**，或者在某个只有黑框框的环境下改点东西。
- 你只要装了 `claude-code`，在终端敲 `claude`，就相当于把一个顶级的 AI 程序员带到了这台服务器上，帮你查日志、写 Bash 脚本、修改 Nginx 配置。

### 3. 深度的上下文理解与项目级重构
Claude Code 能够非常快速地读取和理解你当前终端所在的整个文件夹的代码结构。
- **场景**：你可以让它在后台默默地干一些繁琐的“重构”工作。
- 比如：`claude "把这个项目里所有的 class 组件全部用 React Hooks 重写一遍，并确保测试通过。"`
- 它会像一个不知疲倦的外包小弟一样，一个文件一个文件地改，并自己跑测试验证。

---

### 总结：你该用 Cursor 还是 Claude Code？

*   **用 Cursor (你现在正在用的)**：如果你在**写核心业务逻辑、设计架构、推演测试方案数据**（比如咱们刚才做的），Cursor 的侧边栏对话 + 极其优秀的代码 Diff 对比界面，是目前地球上体验最好的。
*   **用 Claude Code**：当你遇到了**恶心的环境配置问题、跑不通的流水线脚本、几百个文件的枯燥重构**时，去终端里敲下 `claude`，让它在后台自己“折腾”去，折腾好了再叫你。

### 通俗比喻

- **Cursor = “一间全副武装的高科技可视工作室”**
  - **特点**：有窗户（图形界面）、有图纸（文件目录）、有画板（代码编辑区），旁边还站着一个非常聪明的助理（右侧的对话框，也就是现在的我）。
  - **工作模式**：你指着图纸说：“把这面墙改成红色的”。助理马上在图纸上给你画出来，你**亲眼看着**（红绿代码对比 Diff），觉得没问题，点一下“Accept（同意）”，墙才变红。**你在主导，你在审核，所见即所得。**

- **Claude Code = “一个住在地下室（终端 Terminal）的蒙眼金牌电工师傅”**
  - **特点**：没有图形界面，没有按钮，只有一个黑乎乎的命令行窗口。
  - **工作模式**：你在门口喊一声：“师傅，整栋楼停电了，去帮我修一下！”师傅提着工具箱就钻进地下室了。他会**自己**去查电表（看报错日志）、**自己**拉电线（修改代码）、**自己**推电闸测试（跑测试命令），中间修错了就自己再改。直到最后灯亮了，他跑出来跟你说：“修好了！” **他全自动干活，不需要你一步步点同意。**

---

### 核心区别总结

| 区别维度 | Cursor (你现在用的) | Claude Code |
| :--- | :--- | :--- |
| **存在形态** | 庞大的**图形化代码编辑器** (像 VS Code 一样) | 小巧的**命令行工具** (只有黑框框敲字) |
| **交互方式** | **协作式/半自动**：AI 改了代码，你肉眼看对比（Diff），你点确认才生效。 | **代理式/全自动**：你发指令，它自己去查日志、改文件、跑终端命令，一气呵成。 |
| **擅长干什么** | 1. 从零开始写新代码/文档（如咱们刚才写的测试方案）<br>2. 梳理业务逻辑<br>3. 精细化地阅读和修改指定文件 | 1. 解决恶心的环境报错<br>2. 批量重构几百个文件<br>3. 自动运行和修复测试脚本 |
| **工作位置** | 需要在电脑上打开 Cursor 软件界面 | 可以在任何终端里（哪怕是通过 SSH 连上的没有屏幕的 Linux 远程服务器） |

**它们不是替代关系，而是“黄金搭档”。**

### 🤖 AI 智能体 (Assistant)

**RPA（机器人流程自动化，Robotic Process Automation）强行自动**，在现在的 AI 语境下，指的是使用专门的 **Browser Agent（浏览器自动化智能体）** 来代替人手去点击网页、输入文字和提取内容。

在咱们 Cursor 的工具箱里，有一个内置的神器叫做 `Task tool (subagent_type="browser-use")`，它就是做这个的。

---

### RPA 强行自动是怎么工作的？

如果你把网页地址交给我，它的工作流程大概是这样：

1. **阅读 Excel 题目**：我先用 Python 脚本把你 `result.xlsx` 里的 80 个问题提取成一个列表。
2. **召唤浏览器机器人**：我在后台唤醒 Cursor 自带的 `browser-use` 机器人，告诉它你们“经分智能体”网页对话框的网址。
3. **模拟人工操作（核心）**：
   - 机器人会打开一个真实的浏览器（可能在后台运行）。
   - 它能“看懂”网页的结构，找到那个**输入框**的元素，把第一道题复制进去。
   - 找到并点击**“发送”按钮**。
   - **智能等待**：它会不断监测网页，直到发现出现了一段新的文字（也就是智能体的回答停止生成了）。
   - 它把那段长文本**复制**下来。
   - 然后在对话框点击“新建对话”或者清空输入框，准备问下一题。
4. **回填表格**：它把拿到手的结果传回给我，我用 Python 把这段回复精准填进 `result.xlsx` 的“智能体回复”列里。
5. **循环 80 次**，直到所有题目跑完。

---

### RPA 方法的优缺点

| 维度 | 优点 | 缺点 / 痛点 |
| :--- | :--- | :--- |
| **接入成本** | **极低**。完全不需要开发写后端接口，也不管你底层的鉴权逻辑，只要人能在网页上点，机器人就能点。 | 如果网页有极度复杂的**扫码登录**、图形验证码，机器人第一步就会卡住，必须人先手动扫码。 |
| **真实性** | **100% 真实体验**。测出来的效果就是用户在网页上看到的效果（包括前端把 markdown 渲染后有没有吃掉字）。 | 网页结构如果发生微调（比如“发送”按钮换了位置或名字），可能导致机器人找不到元素而报错。 |
| **速度** | 比人手工复制快很多，不用你一直盯着屏幕。 | **远慢于 API 接口测试**。因为它要等网页动画加载、等流式打字效果出完，可能一题就要花十多秒，80题要十几分钟。 |

### 🤖 AI 智能体 (Assistant)

这是一个非常专业且切中要害的测试难点：**大模型（智能体）的输出通常是自由文本，如果不严格按照我们的“三段式”输出，怎么客观、量化地打分？**

我们预设的“深度分析 + 数据支持 + 建议行动”只是**评分基准（Ground Truth）**，是为了让我们（测试人员）心中有一把尺子。在实际阅卷时，我们**不是看它有没有写这三个小标题，而是看它的回答里有没有包含这三个维度的实质内容**。

为了高效、客观地打分，业界目前最主流的解法是引入 **LLM-as-a-Judge（用大模型来当裁判）** 的评分机制，或者**人工多维度切片评分法**。

以下是具体的评分方案和操作方法：

### 方案一：人工多维度切片打分法（适合小批量精细打分）

你可以把一道题的满分设为 **5 分**，然后把我们预设的“三段式基准”拆解成具体的**给分点（Rubric）**，逐一核对智能体的答案：

1. **事实与数据层（占比 40%，即 2 分）**
   - *评判标准*：智能体是否找对了底层数据？有没有算出正确的排名、数值或提取到正确的字段？
   - *打分*：找到了关键实体和准确数据得 2 分；数据找对了一半或存在遗漏得 1 分；完全算错或产生严重“幻觉”（胡编乱造）直接 0 分。
2. **归纳与分析层（占比 30%，即 1.5 分）**
   - *评判标准*：智能体有没有把数据翻译成“人话”？有没有指出问题背后的根因（例如发现“预算不足”是最大痛点）？
   - *打分*：具备业务视角的归纳得 1.5 分；只是像机器一样念数据（“A是5，B是3”）得 0.5 分。
3. **落地与建议层（占比 30%，即 1.5 分）**
   - *评判标准*：有没有针对分析出的痛点，给出类似“跨部门协同”、“调整KPI”或“建立预警”的合理建议？
   - *打分*：给出了具有针对性和可操作性的建议得 1.5 分；给出了一堆车轱辘废话（如“我们要加强管理、提高警惕”）得 0.5 分；没给建议得 0 分。

**最终得分**：将上述三项相加。即使智能体写成了一段连贯的散文，只要文意中覆盖了这三个点，就可以给高分。

### 方案二：LLM-as-a-Judge 自动化打分法（强烈推荐，降维打击）

人工看 80 道开放题的长篇大论会非常疲劳且主观。目前大厂测试团队普遍采用的方法是：**让一个更聪明的模型（比如让 Claude 3.5 或 GPT-4o 作为裁判）来给你们的“经分智能体”打分。**

**具体怎么做？**
我会帮你写一个 Python 脚本，或者你直接在 Cursor 旁边新开一个对话框，输入这样的 **Prompt 裁判指令**：

> **【系统设定】**
> 你是一个资深的企业经营分析总监，现在你需要给实习生（经分智能体）生成的业务分析报告打分。
> 
> **【输入信息】**
> 1. 用户提问：*[把你的开放问题贴进来]*
> 2. 参考基准答案：*[把我之前帮你生成的带有数据和建议的三段式答案贴进来]*
> 3. 智能体实际回答：*[把经分智能体生成的长文本贴进来]*
> 
> **【打分规则（满分 5 分）】**
> 请忽略智能体回答的排版格式（不需要它必须带有“一、深度分析”等标题），纯看语义是否涵盖了基准答案的核心考点：
> - **准确性（2分）**：实际回答中引用的业务数据、数值和客户名称，是否与基准答案一致，有无编造（幻觉）？
> - **洞察力（1.5分）**：实际回答是否指出了基准答案中提到的业务痛点或现象根因？
> - **建议有效性（1.5分）**：实际回答是否给出了与基准答案方向一致或同样具有业务指导意义的管理建议？
> 
> **【输出格式】**
> 1. 得分：[X / 5分]
> 2. 扣分理由：（如果满分则填“涵盖全面，数据精准”）

**这个方法的优势**：
大模型在做语义匹配时非常聪明。哪怕你的智能体把答案写成了一首诗，只要它里面提到了“杨帆签单利润低，建议调整考核”，裁判模型就会自动识别到它命中了“建议层”的得分点，从而给它打分。
